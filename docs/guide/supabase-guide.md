# Supabase SQL 관리 가이드

이 문서는 프로젝트에 포함된 SQL 파일들의 역할과 관리 방법, 그리고 현재 서버 상태를 유지하는 방법에 대해 설명합니다.

## 1. SQL 파일 구분 및 역할

프로젝트의 `supabase/` 디렉토리에는 여러 목적의 SQL 파일이 존재합니다. 상황에 맞는 적절한 파일을 사용하세요.

### 📋 원클릭 설정 (오픈소스용 템플릿)
- **파일**: `supabase/setup_full.sql`
- **목적**: 새로운 Supabase 프로젝트를 생성하고 초기 스키마, 함수, RLS 정책을 한 번에 설정할 때 사용합니다.
- **특징**: 
  - 테이블 생성, 권한 설정, 헬퍼 함수가 모두 포함된 통합 스크립트입니다.
  - 오픈소스 사용자나 로컬 개발 환경 초기화 시 권장됩니다.

### 💾 현재 서버 스키마 (Production/Current)
- **파일**: `supabase/schema.sql`
- **목적**: 현재 실제 운영되고 있는 서버의 최신 DB 구조를 반영합니다.
- **특징**:
  - `supabase db dump` 명령어를 통해 생성된 원본 덤프 파일입니다.
  - 서버의 제약조건, 인덱스, 시스템 설정 등 모든 세부 사항이 포함되어 있습니다.
  - 서버 간 스키마 비교나 복구 시 사용됩니다.

### 🚀 마이그레이션 (증분 변경 내역)
- **디렉토리**: `supabase/migrations/`
- **목적**: 이미 운영 중인 DB에 새로운 컬럼이나 테이블을 추가할 때 사용합니다.
- **특징**:
  - `migration_*.sql` 형식으로 파일이 이름 지어집니다.
  - 기존 데이터를 보존하면서 스키마를 안전하게 변경할 때 사용합니다.

---

## 2. 사용 가이드

### 새 프로젝트에 적용할 때 (초 초기 설정)
1. Supabase SQL Editor를 엽니다.
2. `supabase/setup_full.sql`의 내용을 복사하여 실행합니다.
3. 인증 및 사용자 설정을 진행합니다.

### 서버의 변경 사항을 로컬에 반영할 때
1. `supabase/schema.sql`을 참조하여 서버와 로컬의 차이점을 확인합니다.
2. 최신 `migrations/` 파일들을 순서대로 실행합니다.

---

## 3. 주의 사항
- **중복 주의**: `setup_full.sql`과 `migrations/` 파일의 내용이 중복될 수 있습니다. 운영 중인 서버에는 항상 `migrations/` 파일을 순서대로 적용하는 것이 안전합니다.
- **백업**: 중요한 변경 전에는 항상 `supabase db dump`를 통해 `schema.sql`을 최신화해 두는 것을 권장합니다.
