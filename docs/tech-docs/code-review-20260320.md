# Hanavi Schedule - 프로젝트 코드 리뷰 및 개선 제안 (2026-03-20)

이 문서는 프로젝트의 성능, 에지 케이스(Edge-case) 버그, 구조적 가독성을 높이기 위한 구조적인 코드 리뷰 결과입니다. 이전 UI/UX 및 보안 중심 리뷰를 바탕으로 한 단계 더 심화된 아키텍처 및 로직 리뷰입니다.

---

## 🐛 1. 기능적 버그 및 엣지 케이스 (Critical)

### 🚨 졸업 멤버 과거 스케줄 노출 불가 버그 (`useSchedule.ts`)
- **발견 내용:** `useSchedule.ts`의 `filterGraduates` 함수 내부에서 기준 날짜를 항상 `getMonday(new Date())` (즉, 현실 세계의 현재 주차)로 하드코딩하고 있습니다.
- **문제점:** 사용자가 이전 주차(과거)의 스케줄표를 조회할 때, **그 당시에는 정상적으로 활동했던 멤버라 하더라도 현재 기준으로 졸업 처리가 되어 있다면 과거 스케줄에서도 완전히 사라져버리는 문제**가 발생합니다. 과거의 기록(아카이빙) 목적이 훼손됩니다.
- **조치 제안:** `filterGraduates` 함수가 현재 열려있는 `weekRange` 문자열 값을 받아와서(`getStartDateFromRange` 활용), 조회 중인 해당 주간의 월요일을 기준으로 졸업 여부를 평가하도록 인자를 전달해야 합니다.

## 🏗️ 2. 아키텍처 및 리팩토링 제안 (Architecture & Maintainability)

### 🧹 `src/app/page.tsx` (Home 컴포넌트)의 비대화 해소
- **발견 내용:** `page.tsx`가 480줄에 달하며, 수많은 로직이 하나의 컴포넌트에 섞여 있습니다.
  1. PNG 이미지로 내보내는 매우 복잡한 렌더링/공유 로직 (`handleExport`)
  2. 데스크톱 사이드바 (Drawer) 트리거 및 UI 상태
  3. 모바일 하단 메뉴 (드롭다운) UI 상태
- **문제점:** 메인 페이지의 유지보수가 어려워지고 가독성이 떨어집니다. 로직 수정 시 불필요한 리렌더링이 발생할 수 있습니다.
- **조치 제안:**
  - `handleExport` 전체 로직을 `useExportSchedule`이라는 커스텀 훅으로 분리.
  - 내비게이션 메뉴들(사이드바, 모바일 메뉴)을 `Sidebar.tsx`, `MobileMenu.tsx` 등의 컴포넌트로 완전히 분리.

### 🌬️ 스와이프 제스처 로직 커스텀 훅 분리 (`ScheduleGrid.tsx`)
- **발견 내용:** `ScheduleGrid.tsx` 컴포넌트 내부에 `onTouchStart`, `onTouchMove`, `onTouchEnd` 등의 스와이프 계산 로직이 하드코딩되어 있습니다.
- **조치 제안:** `useSwipe({ onSwipeLeft, onSwipeRight })` 형태의 커스텀 훅(`src/hooks/useSwipe.ts`)으로 분리하여 Grid 컴포넌트의 가독성을 높이고, 추후 다른 요소에서도 제스처 액션을 쉽게 도입할 수 있도록 설계합니다.

## 🎨 3. UI/UX 세부 구현 (Design Implementation)

### 📏 `ScheduleCell.tsx`의 텍스트 복잡도 하드코딩
- **발견 내용:** 스케줄 내용의 길이를 측정해 폰트 크기를 조절하는 `complexityScore` 로직이 존재합니다 (`if (complexityScore > 100) ...`).
- **조치 제안:** 현재 잘 작동하지만 텍스트 길이에 의존하는 하드코딩 방식이므로, 장기적으로는 CSS `line-clamp`나 컨테이너 쿼리(Container Queries) 혹은 `clamp()` 함수를 활용하여 CSS 단에서 유연하게 글씨 크기가 반응형으로 줄어들도록 뷰 로직 이관을 고려해볼 수 있습니다. (당장 수정할 필요는 없으나 기술 부채로 기록)
