# 프로젝트 코드 리뷰 보고서

## 1. 아키텍처 및 코드 구조
### 🚨 관리자 페이지(`AdminPage`) 비대화
- `src/app/admin/page.tsx` 파일이 약 1,580라인에 달하는 거대 컴포넌트로 구성되어 있습니다.
- **문제점**: 인증 로직, 데이터 페칭, 대규모 비즈니스 로직(YouTube 연동), 수많은 모달 상태가 한 파일에 섞여 있어 유지보수가 어렵습니다.
- **개선 제안**:
    - 비즈니스 로직을 커스텀 훅(`useAdminSchedule`, `useYouTubeSync` 등)으로 분리.
    - 각 모달 및 UI 섹션을 독립적인 컴포넌트로 분리.

### ⚠️ 데이터 페칭 일관성 부족
- 메인 페이지는 `useSchedule` (SWR + LocalStorage 캐시)을 사용하지만, 관리자 페이지는 `useEffect` + `fetch`를 직접 사용합니다.
- **개선 제안**: `useSchedule` 훅을 확장하여 관리자 모드에서도 사용할 수 있도록 통합하여 코드 중복을 줄이고 캐싱 전략을 통일.

## 2. 성능 최적화
### ⚡ YouTube 자동 연결 로직 개선
- 현재 프론트엔드에서 루프를 돌며 각 멤버의 영상을 API로 요청하고 있습니다.
- **문제점**: 멤버 수가 늘어날수록 API 할당량 소모가 빠르고 클라이언트 성능에 부담을 줄 수 있습니다.
- **개선 제안**: 백엔드 API Route 또는 Cron Job으로 이관하여 서버 측 캐싱을 강화하고 안정성을 확보.

### 🏗️ 서버 컴포넌트 활용 검토
- 현재 대부분의 컴포넌트가 `"use client"`로 선언되어 있습니다.
- **개선 제안**: 스케줄 그리드의 초기 렌더링을 서버 사이드에서 처리하도록 전환하여 초기 로딩 속도(LCP)를 개선.

## 3. 보안 및 안정성
### 🔒 RLS(Row Level Security) 정책
- 다수의 migration 파일을 통해 RLS 정책이 고도화되고 있으나, `anon` 키로 접근 가능한 범위에 대한 최종 점검이 필요합니다.
- **개선 제안**: `user_roles` 테이블을 통한 권한 제어가 모든 API(특히 `admin/schedule`)에서 엄격히 적용되는지 정기적인 유닛 테스트 도입.

### 🛡️ 에러 핸들링
- 많은 부분에서 `alert()`를 사용하고 있으며, 예외 발생 시 전체 페이지가 멈출 위험이 있습니다.
- **개선 제안**: `React Error Boundary`를 도입하고, `Toast UI`를 통해 사용자에게 부드러운 에러 피드백을 제공.

## 4. UI/UX 및 품질 (백로그 연동)
### ✨ UI 피드백 강화
- 데이터 로딩 시의 레이아웃 흔들림(Layout Shift) 방지를 위한 `Skeleton UI` 도입 필요.
- `alert()` 대신 일관된 디자인의 `Toast` 컴포넌트 도입.
